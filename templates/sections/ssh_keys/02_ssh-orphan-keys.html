Orphan SSH Keys

As SSH keys proliferate within an environment, management of these keys becomes increasingly difficult. Administrators are likely to create new keys so that they are able to do their job effectively, without the distraction of having to re-authenticate for every new connection, and this often means creating keys that allow both workstation-to-server connections as well as server-to-server connections. With no centralised oversight of these key pairs, effective control of these keys is impossible, particularly if a user leaves the organisation - security teams simply do not know which machines this user may or may not have left keys.

This leaves a security hole which can be exploited by attackers. It is possible that a breach may allow access to a Unix server, and if the attacker finds SSH private keys on that server these can be stolen and used to move laterally to other servers. Understanding where these keys currently reside is therefore key to being able to control the risks of lateral movement.
